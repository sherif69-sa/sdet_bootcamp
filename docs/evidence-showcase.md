# Evidence showcase: what SDETKit actually produced in a representative run

**Use this page if:** you want representative real artifact examples from this repository.

**Canonical artifact interpretation is in:** [CI artifact walkthrough](ci-artifact-walkthrough.md).

This page shows a **real run in this repository** (`/workspace/DevS69-sdetkit`) and the concrete files generated by SDETKit.

## Representative scenario

Scenario used for this evidence set:

- Operator runs a fast quality gate, strict security budget enforcement, then release preflight.
- The run is intentionally useful for triage because gates fail and produce actionable structured output.

Commands actually run:

```bash
mkdir -p build
python -m sdetkit gate fast --format json --out build/gate-fast.json
python -m sdetkit security enforce --format json --out build/security-enforce.json --max-error 0 --max-warn 0 --max-info 0
python -m sdetkit gate release --format json --out build/release-preflight.json
```

Observed exit codes in this run:

- `gate fast`: `2`
- `security enforce`: `1`
- `gate release`: `2`

## Artifact files produced

The commands above generated:

```text
build/
├── gate-fast.json
├── security-enforce.json
└── release-preflight.json
```

## What each artifact means

### `build/gate-fast.json`

Purpose:

- Structured result of the fast confidence lane.
- Shows which sub-steps passed/failed (`doctor`, `ci_templates`, `ruff`, `ruff_format`, `mypy`, targeted `pytest`).

Real excerpt from this run:

```json
{
  "failed_steps": [
    "ruff",
    "ruff_format"
  ],
  "ok": false,
  "profile": "fast"
}
```

Operator value:

- You get deterministic failed step IDs without scanning long logs.
- You can jump directly to lint/format remediation when those IDs fail.

### `build/security-enforce.json`

Purpose:

- Security budget decision object with explicit counts, limits, and exceeded metrics.

Real excerpt from this run:

```json
{
  "counts": {
    "error": 0,
    "info": 131,
    "total": 131,
    "warn": 0
  },
  "exceeded": [
    {
      "count": 131,
      "limit": 0,
      "metric": "info"
    }
  ],
  "ok": false
}
```

Operator value:

- Gate reason is machine-readable (`info` budget exceeded), so CI policy decisions are auditable.

### `build/release-preflight.json`

Purpose:

- Release lane aggregation that records preflight step outcomes.
- Shows whether `doctor --release`, playbook validation, and nested `gate fast` passed.

Real excerpt from this run:

```json
{
  "failed_steps": [
    "gate_fast"
  ],
  "ok": false,
  "profile": "release"
}
```

Operator value:

- Explains why release preflight failed without re-running the full workflow interactively.

## What to inspect first when a gate fails

1. Open `build/release-preflight.json` first for top-level release decision and failed step IDs.
2. If `gate_fast` failed, inspect `build/gate-fast.json` and read `failed_steps`.
3. For security policy/budget failures, inspect `build/security-enforce.json` (`counts`, `limits`, `exceeded`).
4. Only after the JSON confirms the failing step, drill into raw logs for full context.

## How to use these in CI/review

- Upload these JSON files as CI artifacts (`ci-gate-diagnostics` / `release-diagnostics` style).
- In pull requests, link to the exact JSON artifact and paste only the key fields (`ok`, `failed_steps`, exceeded budgets).
- Keep artifacts with release records to support audit/review decisions later.
- Use JSON as the source of truth for pass/fail automation; use terminal logs as secondary forensic detail.

## Why this is better than raw terminal logs alone

- JSON artifacts preserve structure (`ok`, `failed_steps`, `counts`, `limits`) for automation.
- They are easier to diff between runs than free-form logs.
- They let reviewers quickly confirm **decision inputs** instead of manually parsing mixed stdout/stderr.

## Limitations / representative example note

- This is a representative run from this repository state and branch, not a universal baseline for all repositories.
- Counts and failing step IDs can differ by project content, policy thresholds, and local tooling state.
- The commands and snippets here are copied from actual generated outputs in this run (no fabricated artifacts).
